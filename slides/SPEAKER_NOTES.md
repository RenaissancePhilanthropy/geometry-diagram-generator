# Speaker notes — `onboarding.html`

These are conversational walkthrough notes you can read aloud or paraphrase while presenting. Open this file on a second monitor (or your phone) and scroll along with the deck. Total spoken time: roughly **10–12 minutes**.

Tone tip: don't read the slide. The slide is the visual; you're the explainer. Use the slide to anchor what you're saying, but always add the "why this matters" that the slide can't fit.

---

## Slide 1 — Title

> *On screen: dark hero, "A benchmark for **generative** geometric reasoning."*

**Say something like:**

"Welcome. Quick orientation: this project is called **GeoGenBench**, and the easiest way to describe it is — it's a benchmark we built to test whether language models can *draw* geometry diagrams correctly from a written description. Like, you tell GPT 'draw an acute triangle with these angles and add the altitude,' and we can automatically grade whether what it produced is right or not.

The point of the next ten minutes is just to give you a map. By the end of this you should know what the project actually does, where things live in the repo, and what to type into your terminal to get a smoke test running today. We'll go fast — this is a map, not a manual. I'll point you at the docs and code that have the depth."

**Pro tip:** if the audience already knows the project at a high level, just spend 10 seconds on this slide and skip to slide 2. If they're brand new, linger here long enough to make eye contact and reassure them they don't need to memorize anything.

---

## Slide 2 — The big picture

> *On screen: light slide, big claim — "We test whether language models can **draw correct geometry diagrams** from a textual description."*

**Say something like:**

"So here's the gap we noticed. There are *tons* of geometry benchmarks out there — but they all do it the *opposite* way. They show a model a diagram and ask 'how big is angle X?' or 'is this triangle isosceles?' That's a *reading* task.

Nobody was testing the *production* task — can the model take a sentence like 'draw a circle with these three points on it' and actually produce a correct figure. Which is weird, because that's the deployed task. When teachers use LLMs to author worksheets, when slide decks get generated, when technical documentation gets illustrated — the LLM is *making* the diagram, not reading one.

So we built the benchmark for that. **Eight hundred prompts, each paired with a machine-checkable rubric** that tells you exactly which clauses must be true for the diagram to count as correct. The trick that makes the whole thing work — and this is the part that's actually novel — is the rubric. We'll get to that on the next slide."

**Things to emphasize:**
- "Generative" is the key word. The model is *producing*, not *interpreting*.
- Machine-checkable, deterministic, no humans. That's the differentiator.
- Don't dwell on prior benchmarks unless asked.

## Slide 3 — Methodology (two pillars)

> *On screen: split panel — top half white "Pillar 1: Procedural co-generation," bottom half dark "Pillar 2: Render-then-symbolic-verify."*

**Say something like:**

"Okay, this is the most important slide. If you remember nothing else, remember these two ideas — they're the whole methodological contribution.

**Pillar one: procedural co-generation.** Imagine you wanted to make 600 geometry problems by hand. You'd write each prompt, then write each grading rubric, and somewhere along the way the rubric and the prompt would drift apart — you'd grade for a 90-degree angle that the prompt actually said was 60 degrees. So we don't write them by hand. Each scenario is generated by a *single Python function* that emits *both* the prompt prose *and* the property-list rubric at the same time, from the same source code. They literally cannot disagree because they came from one source.

That also gives us a free defense against contamination — when reviewers worry the model memorized our benchmark, we just re-roll the engine with a new random seed and produce a fresh split with the same difficulty distribution.

**Pillar two: render-then-symbolic-verify.** When a model gives us TikZ — TikZ is the LaTeX language for drawing diagrams — we *don't* eyeball it, and we *don't* ask another LLM to be the judge. We *compile* the TikZ into actual SymPy geometry objects in Python. Real points, lines, circles, with real coordinates. Then we evaluate the rubric symbolically: 'is angle A 90 degrees? Is point H actually on segment EF?' These are math expressions, decided in microseconds, deterministic. No vision model, no human grader, no LLM-as-judge nonsense.

That's the methodology. Everything else in the codebase exists to make these two ideas work in practice."

**Things to emphasize:**
- "Co-generation" = same source emits both prompt and rubric. Drum that in.
- "Symbolic verification" = SymPy decides, not an LLM. This is the credibility hook.

**Audience may ask:**
- "Why not LLM-as-judge?" → Because LLMs disagree with themselves run-to-run, can be gamed, and are expensive. SymPy gives you pass/fail in microseconds, free, every time the same answer.
- "What if the LLM emits something weird, like SVG instead of TikZ?" → The methodology is renderer-agnostic. We support SVG output too. TikZ is just the default because the LaTeX ecosystem has the best geometry libraries.

## Slide 4 — The pipeline

> *On screen: dark slide, five numbered cards — Prompt → Strategy → Render → Compile → Verify.*

**Say something like:**

"So that's the philosophy. Now here's what actually happens to one scenario, end to end.

We start with a **prompt** — a natural-language sentence describing what to draw. Could be from our procedural template engine, or from the curriculum-extracted set we'll talk about later.

That prompt goes through a **strategy** — that's the LLM call. We have three reference strategies, and we'll cover them on the repo tour. The output is always TikZ — that's the universal currency.

Then **render**: we ship a Docker container running LuaLaTeX that takes TikZ in and gives an SVG out. That's what we'd ship to a user as the rendered diagram.

In parallel, we **compile** — same TikZ, but instead of going to LaTeX, we extract coordinates and instantiate SymPy points, lines, circles. This is where 'render then verify' splits into two paths.

Finally, **verify**: each predicate in the rubric is a Python function that takes the SymPy objects and returns true or false. The scenario gets a `gate_status` — pass, soft pass, fail.

The thing to internalize: steps 4 and 5 *don't need an LLM*. Once the model has produced TikZ, the rest of the pipeline is pure code. That's why this benchmark is essentially free to run repeatedly and why it's reproducible."

**Things to emphasize:**
- "Same TikZ" goes to both render *and* compile. They're not independent paths.
- The cost of grading is essentially zero. Cost is only in step 2.

## Slide 5 — Repo tour

> *On screen: paper slide, six directory cards — `ir/`, `strategies/`, `recipe/`, `evals/`, `renderer/`, `paper/`.*

**Say something like:**

"Let's walk through the codebase. There are six folders you'll spend the most time in, and they map directly to the pipeline.

**`ir/`** — this is the heart of the project. IR stands for 'intermediate representation.' It's the typed data structure that geometry passes through. The Pydantic models are in `ir/ir.py`, the SymPy compiler is in `ir/to_sympy.py`, and all the predicate definitions are in `ir/checks.py`. If you're going to read just one folder, read this one.

**`strategies/`** — three reference implementations of how to ask a model to draw something. `raw_code` is 'just give me TikZ directly.' `structured` is 'fill out this typed JSON form, and we'll compile it.' `recipe` is 'pick from our menu of named constructions.' Same input prompt, three very different things the model has to do.

**`recipe/`** — this one's a small DSL we built. Think of it like a Lego kit of geometric constructions. `triangle()`, `altitude()`, `circumcircle()`. The `recipe` strategy uses these — they get lowered into the IR.

**`evals/`** — the benchmark harness. Where you'll spend most of your time when you're running experiments. `evals/run.py` is the runner; `evals/leaderboard_analyze.py` makes the tables; `evals/leaderboard_plot.py` makes the figures.

**`renderer/`** — the Docker container with LaTeX inside. You don't touch the code in here often, but the container has to be running on port 8001 before any SVG can come out.

**`paper/`** — the NeurIPS paper draft. If you want to know *why* something is built the way it is, the paper is your answer. It's a 35-page LaTeX project that compiles to a single PDF."

**Things to emphasize:**
- IR is the glue. Everything passes through it.
- The renderer being Docker is intentional — it isolates LaTeX from your laptop, which is a huge sanity win.

## Slide 6 — Setup

> *On screen: dark slide, terminal-styled code block with four commands.*

**Say something like:**

"Okay, hands on the keyboard. Four commands and you're up.

`uv sync` installs the Python deps. We use `uv` instead of pip — it's much faster, and the lockfile is committed.

Add an API key. **Anthropic is the cheapest one to start with**, and we have plenty of credit for it as of today. You only need *one* provider's key to start; the eval runner just routes around any provider that's down.

Then start the renderer. It's a Docker container on port 8001. If you don't have Docker installed, get it from docker.com — it's free for individual use. Once it's running you can curl `localhost:8001/health` to confirm.

Last command runs the smoke test — it's called `dry_run` and it picks one scenario from the GenExam benchmark and runs the full pipeline against it. About 30 seconds end-to-end. If that works, your environment is good.

The `DYLD_FALLBACK_LIBRARY_PATH` thing is a macOS quirk — `cairosvg` needs to find the `libcairo` system library, and Homebrew installs it in a place that the Python wrapper doesn't search by default. Linux users skip that prefix."

**Pro tip for live demo:** *actually run* the smoke command on your screen if there's time. Watching the pipeline produce a real SVG is way more convincing than describing it.

> *On screen: light slide, three numbered tasks — read CLAUDE.md, skim paper §3, run dry-run.*

**Say something like:**

"Concrete plan for after this meeting. Three tasks, in this order, and you should be done in about thirty minutes.

**One: read `CLAUDE.md`.** That's the developer brief at the root of the repo. It has every command, the architecture summary, and pointers to everything else. It's *the* canonical map for the codebase. Don't skip it — it'll save you an hour of wandering around later.

**Two: skim the paper, just the intro and section 3.** Pages 1 through 9. Look at Figures 1, 2, and 4 — those are the worked example, the procedural co-generation diagram, and the three-strategies-side-by-side. Together those three figures explain the whole project visually in about five minutes.

**Three: run a dry-run command from the previous slide.** When it finishes, open the SVG it generated. The path is in the script's output. Sanity-check that the pipeline actually produced something that looks like a geometry diagram.

If those three things work, you've onboarded. Everything else is depth."

**Things to emphasize:**
- This is a *checklist*, not a suggestion. They should literally do these three things today.
- Task 3 is the one that catches problems — if the smoke test fails, that's something to surface immediately, not silently struggle with.

## Slide 9 — Common workflows

> *On screen: dark slide, code block with five commands.*

**Say something like:**

"And these are the commands you'll run weekly once you're up to speed.

**`pytest tests/`** — runs the unit tests. We're test-driven on the IR layer especially. Every PR should keep these green.

**`evals.run`** — runs a benchmark cell. The flags here say 'one strategy, one model, three repeats per scenario.' You can dial those up or down. The output is JSONL, one record per scenario per repeat.

**`evals/eval_viewer.py`** — pops up a local web UI for browsing results. Backend on port 8002, frontend on a Vite dev server. Super useful for visually inspecting what each model produced — you can flip through SVGs side-by-side.

**Rebuild the paper** — anytime you edit a `.tex` file, rebuild via the texlive Docker container. Takes about thirty seconds. The PDF lands at `paper/geogenbench.pdf`.

**`evals.regrade`** — this is the magic command for verifier changes. Because we store every model's TikZ output verbatim in the JSONL, we can re-run the verifier against a different commit's check semantics *without spending another dollar on LLM calls*. Saves us hundreds of dollars per audit."

**Things to emphasize:**
- The `regrade` workflow is one of the project's quiet superpowers. Mention it.
- Eval viewer is gold for sanity-checking — strongly recommend they try it on day 1.

## After the deck — anticipated Q&A

A handful of questions you'll probably get. Pre-loaded answers:

| Question | Short answer |
|---|---|
| "How is this different from GenExam?" | GenExam uses an LLM as judge. We use SymPy. Section 2.3 of the paper has the full table comparing us to GenExam, GeoQA, MathVista, etc. |
| "Why is the IR a Pydantic schema and not just JSON?" | Type validation — if a model emits `{"name": 5}` where a string is required, Pydantic rejects it before we waste a SymPy compile cycle. Saves debugging time and gives the model a structured retry signal. |
| "Why three strategies? Isn't `structured` enough?" | We need the comparison to *show* that the IR adds value. `raw_code` is the baseline; `recipe` is the upper bound on scaffolding. Also, `raw_code` is the only strategy compatible with vendors whose constrained-decoding doesn't support our schema (e.g. Gemini Flash). |
| "How long does a full headline run take?" | About 4 hours for one full model column (3 strategies × 2 splits × 3 repeats), running at concurrency 4–8 against the renderer. The chunked runner script in `tmp/` shows how to do it without the SymPy memory leak we hit a few times. |
| "Where do new templates go?" | `evals/scenarios.py` and `evals/scenarios_generated.yaml`. Add a Python function that emits `(prompt, properties)`, register it, regenerate the YAML. There's an example in the existing 30 templates — pick the one closest to what you want and modify. |
| "Can I run this without Docker?" | Sort of. The renderer needs LaTeX, which is much easier in Docker. But `strategies/raw_svg.py` bypasses TikZ + LaTeX entirely and goes straight to SVG via Python. It's not as polished but it works for development without Docker. |
| "Can I add a new model provider?" | Yes — `pydantic-ai` already supports Anthropic, OpenAI, Google, Groq, Cohere, Mistral, OpenRouter. Drop the API key in `.env`, set the model string in `--model`, and you're done. We've used `groq:qwen/qwen3-32b` for an open-weights row, for example. |
