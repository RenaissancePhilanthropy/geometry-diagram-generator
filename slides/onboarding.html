Speaker notesslide 02 / 10 · ~1 min

Here's the gap we noticed. There are tons of geometry benchmarks out there — but they all do it the opposite way. They show a model a diagram and ask “how big is angle X?” or “is this triangle isosceles?” That's a reading task.

Nobody was testing the production task — can the model take a sentence like “draw a circle with these three points on it” and actually produce a correct figure. Which is weird, because that's the deployed task. When teachers use LLMs to author worksheets, when slide decks get generated, when technical documentation gets illustrated — the LLM is making the diagram, not reading one.

So we built the benchmark for that. Eight hundred prompts, each paired with a machine-checkable rubric that tells you exactly which clauses must be true for the diagram to count as correct. The trick that makes the whole thing work — and this is the part that's actually novel — is that rubric. We'll get to it on the next slide.

Emphasize“Generative” is the key word: the model is producing, not interpreting. Machine-checkable, deterministic, no humans — that's the differentiator.

03 / 10
Speaker notesslide 03 / 10 · ~2 min · the most important slide

Okay — this is the most important slide. If you remember nothing else, remember these two ideas. They're the whole methodological contribution.

Pillar 1Procedural co-generation. Imagine you wanted to make 600 geometry problems by hand. You'd write each prompt, then write each grading rubric, and somewhere along the way they'd drift apart — you'd grade for a 90° angle that the prompt actually said was 60°. So we don't write them by hand. Each scenario is generated by a single Python function that emits both the prompt prose and the property-list rubric, from the same source code. They literally cannot disagree because they came from one source. As a bonus: when reviewers worry the model memorized our benchmark, we just re-roll the engine with a new seed and produce a fresh split.

Pillar 2Render-then-symbolic-verify. When a model gives us TikZ — TikZ is the LaTeX language for drawing diagrams — we don't eyeball it, and we don't ask another LLM to be the judge. We compile the TikZ into actual SymPy geometry objects: real points, lines, circles, with real coordinates. Then we evaluate the rubric symbolically: “is angle A 90°? Is point H actually on segment EF?” Math expressions, decided in microseconds, deterministic, free.

Likely Q“Why not LLM-as-judge?” → LLMs disagree with themselves run-to-run, can be gamed, and are expensive. SymPy gives pass/fail in microseconds, free, every time the same answer.

04 / 10
Speaker notesslide 04 / 10 · ~1.5 min

That was the philosophy. Now here's what actually happens to one scenario, end to end.

We start with a prompt — a natural-language sentence describing what to draw. It can come from our procedural template engine, or from the curriculum-extracted set.

That goes through a strategy — that's the LLM call. We have three reference strategies (next-next slide). The output is always TikZ — TikZ is our universal currency.

Then render: we ship a Docker container running LuaLaTeX that takes TikZ in and gives an SVG out. That's what we'd ship to a real user as the rendered diagram. In parallel, compile: same TikZ, but instead of going to LaTeX, we extract coordinates and instantiate SymPy points, lines, circles. Finally verify: each predicate in the rubric is a Python function that takes the SymPy objects and returns true or false.

InternalizeSteps 4 and 5 don't need an LLM. Once the model has produced TikZ, the rest of the pipeline is pure code. That's why this benchmark is essentially free to run repeatedly, and why it's reproducible.

05 / 10
Speaker notesslide 05 / 10 · ~2 min

Six folders that map directly to the pipeline.

  • ir/ — the heart of the project. IR = “intermediate representation,” the typed data structure that geometry passes through. Pydantic models in ir/ir.py; SymPy compiler in ir/to_sympy.py; predicates in ir/checks.py. If you read just one folder, read this one.
  • strategies/ — three ways to ask a model to draw. raw_code is “just give me TikZ.” structured is “fill out this typed JSON form, we'll compile it.” recipe is “pick from our menu of named constructions.” Same prompt, three very different things to do.
  • recipe/ — small DSL we built. Lego kit of geometric constructions: triangle(), altitude(), circumcircle(). Lowers into the IR.
  • evals/ — the benchmark harness. run.py is the runner; leaderboard_analyze.py makes the tables; leaderboard_plot.py makes the figures.
  • renderer/ — the Docker container with LaTeX. Has to be running on port 8001 before any SVG can come out.
  • paper/ — 35-page LaTeX project. If you want the why, the paper is your answer.

EmphasizeIR is the glue — everything passes through it. Renderer being Docker is intentional: it isolates LaTeX from your laptop, which is a huge sanity win.

06 / 10
Speaker notesslide 06 / 10 · ~1 min · hands on keyboard

Four commands and you're up.

uv sync installs the Python deps. We use uv instead of pip — it's much faster, and the lockfile is committed.

Add an API key. Anthropic is the cheapest one to start with, and we have plenty of credit for it as of today. You only need one provider to start; the runner just routes around any provider that's down.

Then start the renderer — Docker container on port 8001. If you don't have Docker, get it from docker.com (free for individuals). Once it's up, curl localhost:8001/health confirms.

Last command runs the smoke test — dry_run picks one scenario and runs the full pipeline against it. About 30 seconds end-to-end. If that works, your environment is good.

TipActually run this on screen if there's time. Watching a real SVG come out is way more convincing than describing it. The DYLD_FALLBACK_LIBRARY_PATH thing is a macOS quirk — Linux skips it.

COMMON WORKFLOWS
Five commands you'll run weekly

Workflows.

# test
pytest tests/

# run an eval cell
uv run python -m evals.run \
    --strategies structured --model anthropic:claude-sonnet-4-6 \
    --repeats 3 --output evals/results

# browse results visually
uv run python evals/eval_viewer.py

# rebuild paper
docker run --rm -v "$(pwd):/work" -w /work \
    texlive/texlive:latest latexmk -lualatex geogenbench.tex

# re-grade existing runs · no new LLM calls
uv run python -m evals.regrade --input-dir evals/results/<run>
09 / 10
Speaker notesslide 09 / 10 · ~1 min · weekly commands

These are the commands you'll run weekly once you're up to speed.

pytest tests/ — runs the unit tests. We're test-driven on the IR layer especially. Every PR keeps these green.

evals.run — runs a benchmark cell. Flags here say “one strategy, one model, three repeats per scenario.” Output is JSONL, one record per scenario per repeat.

eval_viewer.py — pops up a local web UI for browsing results. Backend on port 8002, frontend on a Vite dev server. Super useful for visually inspecting what each model produced — flip through SVGs side by side.

Rebuild the paper — anytime you edit a .tex file, rebuild via the texlive Docker container. Takes about thirty seconds.

evals.regrade — magic command for verifier changes. Because we store every model's TikZ verbatim in the JSONL, we can re-run the verifier against a different commit's check semantics without spending another dollar on LLM calls. Saves us hundreds per audit. TipThis is one of the project's quiet superpowers — make sure to mention it.
