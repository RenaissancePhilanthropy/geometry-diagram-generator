# Learning Curves: Geometry Diagram System — Onboarding Guide

This document is for new team members joining the geometry diagram project. It focuses on **why things are built the way they are** — the design decisions, the approaches we've tried, and what we've learned. For how to run the system, see `CLAUDE.md` in the repository.

---

## What This System Does

The goal is to enable an AI tutor to generate accurate, pedagogically useful geometry diagrams on demand — for example, a student mid-session with LiveHint AI says "can you draw a diagram of this problem so I can see it?" and gets back a correct, legible diagram they can then ask questions about.

A secondary use case (of interest to Carnegie Learning but not our primary focus) is offline batch generation of diagrams for problem sets and textbooks.

The longer-term vision enabled by our architecture: because we resolve geometry symbolically (via SymPy), the system already supports answering student questions like "are angles ABC and DEF equal?" by querying the computed geometry directly, rather than relying on the LLM to reason about it. This is integrated into chat.

---

## The Core Problem

Getting LLMs to produce reliable geometry diagrams is harder than it might look. The challenge isn't just syntax — it's **geometric intuition**. LLMs struggle with things like:

- Knowing which vertex order to use when marking an angle
- Constructing figures that are geometrically valid and non-degenerate (e.g., similar triangles that don't collapse into a line)
- Keeping constraints mutually consistent across a multi-step construction
- Producing diagrams that match a specific visual intent, not just a technically correct interpretation

This has driven almost every major architectural decision: we keep adding structure and scaffolding to compensate for the parts the LLM can't reliably handle on its own.

## The Arc of Approaches

### 1. JSXGraph

Our first target was JSXGraph, a JavaScript library for interactive geometry rendered in the browser. The LLM could write JSXGraph code, but producing *correct* output required things like computing vertex orders to mark the right angle — logic that had to be embedded in handwritten JavaScript helpers rather than generated by the model. This felt like the wrong abstraction level, and interactivity added complexity without enough payoff at this stage.

**Lesson:** The abstraction level matters. Asking the LLM to write low-level code for visual geometry puts too much geometric burden on it.

### 2. Penrose

Penrose is a declarative constraint-based geometry system: you describe geometric relationships symbolically and a layout engine figures out positions. This seemed promising because it aligns well with how geometry is actually defined. The LLM could write Penrose code reasonably well.

The problem was on our side: building reliable non-LLM infrastructure around Penrose (validation, rendering, error recovery) was hard, and even with good infrastructure the output wasn't consistent enough. Subtle issues like constraint sets that technically satisfy the spec but produce degenerate or unexpected diagrams were common. Penrose is now removed from the codebase.

**Lesson:** Even with good LLM output, you need a reliable execution layer. Declarative constraint solvers can produce valid-but-wrong diagrams that are hard to catch automatically.

### 3. TikZ — Raw Code

Moving to TikZ was a significant improvement. TikZ has substantial representation in LLM training data, so models write it fluently. Rendering is handled server-side via a Docker container running LuaTeX, which returns SVG. This also made validation tractable: you can test the SVG directly.

Raw TikZ worked well for straightforward diagrams. For more complex ones — especially those requiring correct angle annotations — it still struggled, though asking the model to self-check often helped. The key limitation is that raw TikZ generation gives you no programmatic handle on the geometry: you can't query "what is the measure of this angle?" without parsing the diagram back out.

**Lesson:** TikZ is a solid rendering backend. But raw code generation doesn't scale to reliable, queryable geometry.

### 4. Structured IR (Intermediate Representation)

To give the system a programmatic handle on geometry, we introduced a structured IR: a JSON schema that the LLM fills in, which is then executed deterministically. This decouples geometric reasoning (done by our code, via SymPy) from diagram description (done by the LLM).

The IR worked, but had expressiveness limitations — some things were awkward or impossible to represent cleanly. This led to the DSL.

**Lesson:** Structured output gives you correctness guarantees the LLM alone can't provide. The tradeoff is expressiveness.

### 5. Tool-Based / Progressive Approach

We experimented with a multi-step tool-calling strategy where the LLM progressively builds a diagram by selecting and invoking tools (add a circle, add a tangent line, etc.). The idea was to give the model more granular control and visibility.

In practice this was too slow and token-heavy for a live tutoring context, and didn't improve diagram quality enough to justify the cost. This approach is currently deprioritized.

**Lesson:** Fine-grained tool use increases latency significantly. For live chat, single-shot generation is preferable.

### 6. DSL + Recipes (Current Approach)

The current best approach combines a custom DSL with a recipe system. The DSL is a JSON-based construction language with structured fields for each step; it gets **lowered deterministically to the IR**, reusing the SymPy → TikZ → SVG pipeline. The DSL was designed to fix expressiveness limitations of the IR while keeping the determinism and testability benefits.

Recipes are YAML files that teach the LLM how to construct specific diagram types. Each recipe includes a worked example construction, context notes, and non-obvious caveats. They function like agent skills: injected into the LLM's context to give it new capabilities it wouldn't reliably have on its own. The recipe catalog is still being built out.

This approach is faster and more reliable than everything we've tried, though it's not perfect — "perfect" is hard with LLMs.

## Current Architecture

Here is the full pipeline for the DSL+recipes strategy:

```
Problem text
    │
    ▼
┌─────────────────────────────┐
│ 1. Recipe selection         │  Small model picks from catalog of
│    (small model, <500 tok)  │  recipe names + one-line descriptions.
└─────────────┬───────────────┘  Outputs: selected recipe IDs,
              │                  unmatched_concepts[], confidence.
              ▼
┌─────────────────────────────┐
│ 2. Prompt assembly          │  System prompt + DSL docs +
│    (deterministic)          │  2–3 fixed worked examples +
└─────────────┬───────────────┘  retrieved recipe YAML skills.
              │
              ▼
┌─────────────────────────────┐
│ 3. Construction generation  │  Main model reads problem + assembled
│    (main model, single call)│  context, outputs JSON construction
└─────────────┬───────────────┘  sequence + annotation block.
              │
              ▼
┌─────────────────────────────┐
│ 4. Execution + validation   │  SymPy resolves all operations.
│    (deterministic)          │  Checks geometric properties.
└─────────────┬───────────────┘  Flags degenerate/impossible constructions.
              │
              ▼
┌─────────────────────────────┐
│ 5. Rendering                │  Converts resolved geometry + annotations
│    (deterministic)          │  to TikZ → SVG (or other backend).
└─────────────────────────────┘
```

**Key design principle: the system is modular.** The rendering backend (currently TikZ → SVG), the generation strategy, and the output format are all swappable. The SymPy layer is what makes the pipeline backend-agnostic: once geometry is resolved to coordinates, it can be rendered many ways.

**Output:** SVG, rendered server-side. This is what enables automated geometric validation (checking lengths, angles, etc. on the resolved geometry before anything is sent to the client).

## The DSL and IR

The system uses two internal representation layers, which are easy to conflate but serve distinct roles.

The **DSL** (domain-specific language) is what the LLM writes during step 3. It's a JSON construction language consisting of an ordered list of **ops** (operations), each with structured fields. Example op types include `triangle`, `circle`, `parallel`, `intersection`, `altitude`, `circumcircle`, and many more. Some ops are "composite" — they expand into multiple lower-level definitions during lowering. The full reference is in `DSL_DOCS` in `recipe/catalog.py`, which is also what gets injected into the LLM's prompt.

The **IR** (intermediate representation) is a lower-level JSON schema that the DSL gets deterministically compiled down to. The IR maps more directly to primitive geometric objects and is what SymPy and the renderer actually operate on. The LLM never writes IR directly.

This two-layer design means:

- The LLM works at a higher, more expressive level (DSL)
- All geometric computation happens in SymPy, not in the LLM
- Validation and rendering operate on a stable, well-defined structure (IR)
- The DSL can evolve without changing the SymPy or rendering layers

## Recipes in Detail

A recipe is a YAML file with these fields:

- **`name`, `description`, `tags`** — used by the selection model to match recipes to requests
- **`context`** — a one-sentence prose description of the givens; sent to the LLM
- **`example`** — a worked construction in DSL format; sent to the LLM as a few-shot example
- **`notes`** — non-obvious caveats and constraints; sent to the LLM
- **`setup`** — prerequisite objects for the construction; **NOT sent to the LLM** — used only for test validation

That last point is important: `setup` exists so our tests can validate the diagram against known geometric truths, without giving the LLM a shortcut to the answer.

Here is a condensed example (parallel lines cut by a transversal):

```yaml
name: parallel_transversal
description: "Two parallel lines AB and CD cut by a transversal EF"
tags: [parallel, transversal, angle, intermediate]
context: "Given points A, B on the first line and C, D on the second line."
example:
  mode: grid
  construction:
    - op: line_through
      id: L1
      points: [A, B]
    - op: parallel
      id: L2
      to_line: L1
      through: C
    - op: point
      id: E
      coords: [2, -1]
      visible: false
    - op: point
      id: F
      coords: [3, 3]
      visible: false
    - op: line_through
      id: Trans
      points: [E, F]
    - op: intersection
      id: G
      of: [L1, Trans]
    - op: intersection
      id: H
      of: [L2, Trans]
  annotations:
    marks:
      - kind: mark_angle
        a: B
        vertex: G
        b: H
        group: 1
notes:
  - "Angle pairs must mark BOTH intersections with the same group number"
  - "Transversal guide points should be visible: false"
```

The catalog of recipes is actively growing. `unmatched_concepts` in the selection output (concepts in a request that no recipe covers) is currently logged but is intended to become a signal for where new recipes are needed — and potentially a basis for deciding when a request is out of scope.

## Chat-Based Editing

Editing works well and was surprisingly easy to implement. Users can ask for changes to an existing diagram conversationally ("make the triangle equilateral", "add the altitude from A") and the system updates incrementally. You can even build a diagram entirely from scratch through a back-and-forth conversation.

One current gap: in the DSL+recipes strategy, editing does not refresh which recipes are in context. This means a follow-up request that would benefit from a recipe not in the original selection may not have access to it.

---

## Styling and Annotations

This is a known weak area, though some progress has been made. Currently supported: angle marks, basic labels, and limited styling such as colored sides. What's still needed for the tutoring use case: arrow/highlight annotations (so the system can say "look at angle ABC" and make it visually obvious) and dimension/length labels.

The tradeoff here is inherent to the structured approach: the DSL/IR constrain what annotations are possible, which prevents the LLM from producing arbitrary or broken TikZ but also limits expressiveness. The "raw TikZ" strategy can render anything TikZ supports, at the cost of less reliable output and no programmatic geometry access.

---

## Evals

Evals exist and include:

- **Did it render?** (less interesting with DSL/IR since the deterministic lowering makes render failures rare; was a real signal with raw TikZ)
- **Geometric property checks** via SymPy (e.g. are these lines actually parallel? is this angle actually 90°?)
- **LLM-as-a-judge** for visual and pedagogical quality

Known limitations: evaluation leans heavily on manual inspection, and it's not clear the collected metrics are the right ones for driving improvement. Evals across different models and prompts haven't been explored thoroughly — this is likely a high-value area. Recipe selection also needs its own dedicated eval, especially as the catalog grows. Instruction-based editing is also not well covered by current evals and should be a focus area.

---

## Open Questions and Known Gaps

- **Styling/annotation support** is the most important near-term gap for the live tutoring use case
- **Recipe selection eval** is needed to ensure the right recipes are retrieved as the catalog scales
- **Recipe context in editing** — the DSL+recipes strategy doesn't refresh recipe context on follow-up edits
- **Complex diagrams** (multi-constraint, multi-shape) remain less reliable
- **`unmatched_concepts`** is logged but not yet acted on — hooking this up to scope detection and catalog gap analysis is future work
- **Eval metrics** — what we're measuring may not be what drives improvement; this is worth revisiting before investing heavily in expanding evals
